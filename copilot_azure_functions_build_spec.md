### shared/models.py

Pydantic models used across all functions.

```python
# Define these models:

class CleaningConfig:
    """Loaded from bhp_config.json. Contains schema, taxonomy, vendors, distributions."""
    # Fields: schema (list of field definitions), category_taxonomy (nested dict),
    # top_20_suppliers (list), units (list), payment_terms (list), etc.
    # Parse directly from the JSON structure in bhp_config.json

class ProfileResult:
    """Output of profile_data function."""
    total_rows: int
    total_columns: int
    columns: list[ColumnProfile]  # one per column
    duplicate_count: int
    near_duplicate_count: int
    encoding_issues: int
    overall_quality_score: float  # 0-1

class ColumnProfile:
    """Profile for a single column."""
    name: str
    dtype_detected: str  # "string", "numeric", "date", "categorical", "mixed"
    null_count: int
    null_rate: float
    unique_count: int
    cardinality: float  # unique / total
    sample_values: list[str]  # 5 representative values
    anomaly_flags: list[str]  # e.g. "mixed date formats", "encoding issues detected"
    distribution_summary: dict  # for numeric: min/max/mean/std, for categorical: top 10 values with counts

class CleaningStrategy:
    """Output of Stage 2 strategy planning (generated by Copilot Studio LLM)."""
    deterministic_fields: list[str]  # fields to clean with rules
    llm_fields: list[str]  # fields needing LLM reasoning
    priority_order: list[str]  # which fields to clean first
    estimated_llm_records: int
    notes: str

class CleanedRecord:
    """A single cleaned record with provenance."""
    record_id: str
    fields: dict  # field_name -> cleaned value
    changes: list[FieldChange]
    confidence: float  # overall 0-1
    needs_review: bool

class FieldChange:
    """Tracks a single field modification."""
    field: str
    original_value: str | None
    new_value: str | None
    method: str  # "regex", "fuzzy_match", "lookup", "llm", "null_injection"
    confidence: float
    agent: str  # which cleaning component made the change

class ValidationResult:
    """Output of validate_output function."""
    is_valid: bool
    total_records: int
    schema_errors: list[dict]  # {record_id, field, error_type, details}
    consistency_issues: list[dict]  # {description, affected_records, severity}
    outliers: list[dict]  # {record_id, field, value, reason}
    stats: dict  # summary statistics

class FlaggedRecord:
    """A record that needs LLM reasoning or human review."""
    record_id: str
    original_row: dict
    partially_cleaned: dict  # deterministic fixes already applied
    flagged_fields: list[str]  # which fields couldn't be resolved
    vendor_candidates: list[dict] | None  # top fuzzy match candidates if vendor is flagged
    suggested_category: dict | None  # best-guess category with confidence
    reason: str  # why it was flagged
```

### shared/date_normaliser.py

Normalises dates from any format to ISO 8601 (YYYY-MM-DD).

```python
def normalise_date(value: str, locale_hint: str = "AU") -> tuple[str | None, float]:
    """Parse a date string in any format and return (ISO date string, confidence).

    Must handle at minimum:
    - YYYY-MM-DD (already clean)
    - DD/MM/YYYY, DD-MM-YYYY (AU standard)
    - MM/DD/YYYY, MM-DD-YY (US format — flag as ambiguous if day ≤ 12)
    - D-Mon-YY, D Mon YYYY ("5-Jan-25", "5 January 2025")
    - YYYYMMDD (no separators)
    - YYYY-MM-DD HH:MM:SS (strip time component)
    - DD/MM (missing year — return None, confidence 0)
    - Excel serial dates (numeric like 45678)

    locale_hint determines ambiguity resolution: "AU" means DD/MM/YYYY is preferred.
    Confidence: 1.0 for unambiguous, 0.7 for ambiguous (e.g. 03/04/2025), 0.0 for unparseable.

    Uses python-dateutil with dayfirst=True for AU locale."""

def normalise_date_column(series: pd.Series, locale_hint: str = "AU") -> pd.DataFrame:
    """Batch normalise an entire column. Returns DataFrame with columns:
    'normalised_date', 'confidence', 'original_format_detected'"""
```

### shared/vendor_matcher.py

Fuzzy matches vendor names against a master list.

```python
class VendorMatcher:
    """Matches dirty vendor names to canonical vendor master list.

    Uses a two-stage approach:
    1. TF-IDF pre-filtering to find candidate matches (fast, narrows search space)
    2. Token-sort ratio from rapidfuzz for final scoring

    Also checks the vendor_dictionary (learned mappings from previous runs) first
    for instant resolution of known variants."""

    def __init__(self, vendor_master: list[dict], vendor_dictionary: dict = None):
        """vendor_master: list of {supplier_name, supplier_id, ...} from config
        vendor_dictionary: dict of {dirty_name: canonical_name} from SharePoint"""

    def match(self, dirty_name: str) -> tuple[dict | None, float, list[dict]]:
        """Match a single vendor name.
        Returns: (best_match_vendor_dict_or_None, confidence, top_3_candidates)

        confidence >= 0.90: auto-resolve (return matched vendor)
        0.70 <= confidence < 0.90: flag for LLM review (return candidates)
        confidence < 0.70: flag as unknown (return candidates anyway)

        Check vendor_dictionary first for instant resolution.
        Then TF-IDF pre-filter to top 20 candidates.
        Then rapidfuzz token_sort_ratio for final scoring."""

    def match_column(self, series: pd.Series) -> pd.DataFrame:
        """Batch match entire column. Returns DataFrame with:
        'matched_name', 'matched_id', 'confidence', 'candidates_json', 'method'
        method is one of: 'dictionary', 'fuzzy_auto', 'fuzzy_flagged', 'unknown'"""
```

### shared/deduplicator.py

Detects exact and near-duplicate records.

```python
class Deduplicator:
    """Detects duplicate records using exact matching and fuzzy near-duplicate detection.

    Exact duplicates: identical on all key fields (date, amount, supplier, invoice_number)
    Near-duplicates: same supplier + similar amount (±5%) + similar date (±5 days) + similar description"""

    def __init__(self, key_fields: list[str] = None):
        """key_fields default: ['date', 'amount', 'supplier_name', 'invoice_number']"""

    def find_duplicates(self, df: pd.DataFrame) -> tuple[list[list[int]], list[list[int]]]:
        """Returns (exact_duplicate_groups, near_duplicate_groups).
        Each group is a list of row indices that are duplicates of each other.
        The first index in each group is the 'canonical' record to keep."""

    def deduplicate(self, df: pd.DataFrame) -> tuple[pd.DataFrame, pd.DataFrame]:
        """Remove duplicates. Returns (deduped_df, removed_df).
        removed_df contains all removed records with a 'duplicate_of' column
        pointing to the kept record's record_id."""
```

## Azure Functions

### 1. profile_data/

**Route:** POST `/api/profile-data`

**Input body:**
```json
{
  "blob_url": "https://storage.blob.core.windows.net/scp-cleaning/input_file.parquet",
  "config": { ... }  // or "config_blob_url" to load from blob
}
```

**Logic:**
1. Download DataFrame from blob_url using blob_helpers
2. Run column-by-column profiling:
   - Detect dtype for each column (string/numeric/date/categorical/mixed)
   - Count nulls, uniques, compute cardinality
   - For numeric: min, max, mean, std, percentiles
   - For string: length stats, detect encoding issues (chardet), sample values
   - For date columns: detect format variants present
3. Run duplicate detection (Deduplicator.find_duplicates, count only — don't remove yet)
4. Compute overall quality score: weighted average of (1 - null_rate) across required fields
5. Return ProfileResult as JSON

**Output body:**
```json
{
  "profile": {
    "total_rows": 51500,
    "total_columns": 24,
    "columns": [ { "name": "supplier_name", "dtype_detected": "string", "null_count": 0, ... } ],
    "duplicate_count": 1500,
    "near_duplicate_count": 1000,
    "encoding_issues": 2500,
    "overall_quality_score": 0.72
  }
}
```

**Timeout:** 60 seconds. At 50K rows, profiling should complete in <10s.

**Tests (test_profile_data.py):**
- Load first 1000 rows of bhp_dirty_50k.xlsx, run profiling, assert columns detected correctly
- Assert duplicate_count > 0 (we know dirty data has injected duplicates)
- Assert encoding_issues > 0
- Assert null counts for nullable fields are > 0

### 2. clean_deterministic/

**Route:** POST `/api/clean-deterministic`

**Input body:**
```json
{
  "blob_url": "https://...",
  "config": { ... },
  "strategy": { "deterministic_fields": ["date", "amount", "supplier_name", "unit", "description"], "llm_fields": ["category_l1", "category_l2", "category_l3"] },
  "vendor_dict_url": "https://sharepoint.../vendor_dictionary.json",
  "abbrev_dict_url": "https://sharepoint.../abbreviation_dictionary.json"
}
```

**Logic (in this order):**
1. Download DataFrame from blob_url
2. Download vendor dictionary and abbreviation dictionary from SharePoint URLs (or blob)
3. **Encoding fix** — run encoding_fixer on all string columns. Log changes.
4. **Date normalisation** — run date_normaliser on 'date' column. Log changes.
5. **Amount normalisation** — run amount_normaliser on 'amount', 'unit_price', 'amount_usd' columns. Log changes.
6. **Unit standardisation** — run unit_standardiser on 'unit' column. Log changes.
7. **Vendor matching** — run vendor_matcher on 'supplier_name' column using vendor master from config + vendor dictionary from SharePoint. Log changes.
8. **Deduplication** — run deduplicator. Remove exact duplicates. Flag near-duplicates for review.
9. **Separate** into two DataFrames:
   - `cleaned`: records where ALL fields have confidence >= 0.70
   - `flagged`: records where ANY field has confidence < 0.70, or vendor match is uncertain, or near-duplicate
10. For each record, build a changes list (FieldChange objects) tracking every modification
11. Upload cleaned and flagged DataFrames to blob storage
12. Return stats and blob URLs

**Output body:**
```json
{
  "cleaned_blob_url": "https://.../cleaned_batch.parquet",
  "flagged_blob_url": "https://.../flagged_batch.parquet",
  "changes_blob_url": "https://.../changes_log.parquet",
  "stats": {
    "input_rows": 51500,
    "cleaned_rows": 41200,
    "flagged_rows": 8800,
    "duplicates_removed": 1500,
    "fields_modified": {
      "date": 9800,
      "amount": 7200,
      "supplier_name": 14500,
      "unit": 9800,
      "description": 1200
    }
  }
}
```

**Timeout:** 230 seconds (HTTP max). For 50K rows with fuzzy matching, target <120s. If >50K rows, the calling agent should chunk the file and call multiple times.

**Performance notes:**
- Vendor matching is the bottleneck. Pre-compute TF-IDF matrix once, reuse for all rows.
- Use pandas vectorised operations where possible, avoid iterrows.
- The vendor_dictionary lookup is O(1) and should resolve 30-50% of vendors instantly on mature runs.

**Tests (test_clean_deterministic.py):**
- Load 1000 rows of dirty data, run cleaning, verify date column is all ISO format
- Verify known vendor typos resolved (e.g. "Caterpilalr" → "Caterpillar Inc")
- Verify amounts are all numeric floats after cleaning
- Verify duplicate count matches expected (dirty data has 3% duplicates)
- Run on full 50K dataset, verify completes within 120s

### 3. classify_categories/

**Route:** POST `/api/classify-categories`

**Input body:**
```json
{
  "records": [
    { "record_id": "BHP-PO-0001234", "description": "Hydraulic hose assembly for Cat 789", "supplier_name": "Caterpillar Inc", "amount": 1500.00, "unit": "EA" },
    ...
  ],
  "taxonomy": { "Equipment & Parts": { "Mobile Equipment": ["Haul truck parts", ...], ... }, ... },
  "examples": [
    { "description": "Replacement track chain for Komatsu PC5500", "category": "Equipment & Parts > Mobile Equipment > Excavator parts" },
    ...
  ]
}
```

**Logic:**
This function handles DETERMINISTIC category classification for obvious cases. It does NOT call an LLM.

1. For each record, attempt rule-based classification:
   - Keyword matching against category taxonomy (e.g. "diesel" → Energy & Fuel > Diesel)
   - Supplier-based inference (Caterpillar → Equipment & Parts > Mobile Equipment)
   - Description similarity to few-shot examples (TF-IDF cosine similarity)
2. Assign confidence score:
   - Keyword match + supplier match + example similarity all agree → 0.95
   - Two of three agree → 0.80
   - Only one signal → 0.60
   - No signal → 0.30
3. Records with confidence >= 0.80 get auto-classified
4. Records with confidence < 0.80 are returned as "needs_llm": true

**Output body:**
```json
{
  "classifications": [
    { "record_id": "BHP-PO-0001234", "category_l1": "Equipment & Parts", "category_l2": "Mobile Equipment", "category_l3": "Haul truck parts", "confidence": 0.92, "needs_llm": false },
    { "record_id": "BHP-PO-0001235", "category_l1": null, "category_l2": null, "category_l3": null, "confidence": 0.35, "needs_llm": true, "best_guess": "Raw Materials & Consumables > Chemical Reagents > Sulphuric acid" },
    ...
  ],
  "stats": { "auto_classified": 850, "needs_llm": 150, "total": 1000 }
}
```

**Timeout:** 60 seconds.

**Tests:**
- Known descriptions should classify correctly: "Bulk diesel delivery" → Energy & Fuel > Diesel
- Ambiguous descriptions should be flagged: "Parts" → needs_llm: true
- Verify supplier context improves classification (Orica + "delivery" → Raw Materials > Chemical Reagents)

### 4. validate_output/

**Route:** POST `/api/validate-output`

**Input body:**
```json
{
  "blob_url": "https://.../cleaned_output.parquet",
  "config": { ... }
}
```

**Logic:**
1. Download cleaned DataFrame from blob
2. Run schema_validator.validate_dataframe — checks types, formats, required fields, taxonomy membership
3. Run consistency_checker.check_consistency — checks cross-record logic
4. Run outlier detection:
   - Amount outliers: z-score > 3 within each category_l1
   - Date outliers: future dates, dates outside config range
   - Quantity outliers: negative quantities, extremely large values
5. Compile all results into ValidationResult

**Output body:**
```json
{
  "is_valid": false,
  "total_records": 50000,
  "error_count": 47,
```

### 5. format_output/

**Route:** POST `/api/format-output`

**Input body:**
```json
{
  "blob_url": "https://.../cleaned_output.parquet",
  "format": "xlsx",
  "include_changes_log": true,
  "changes_blob_url": "https://.../changes_log.parquet"
}
```

**Logic:**
1. Download cleaned DataFrame from blob
2. If format is "xlsx":
   - Write to Excel with openpyxl
   - Sheet 1 "Cleaned Data": all cleaned records, professional formatting
   - Sheet 2 "Changes Log" (if include_changes_log): what changed per record
   - Sheet 3 "Summary": row counts, field modification counts, quality score
3. If format is "csv": simple CSV output
4. Upload formatted file to blob storage
5. Generate SAS URL for download (24hr expiry)

**Output body:**
```json
{
  "download_url": "https://storage.blob.core.windows.net/scp-cleaning/output_cleaned.xlsx?sv=...",
  "expires": "2026-04-09T10:00:00Z",
  "format": "xlsx",
  "file_size_mb": 7.2
}
```

**Timeout:** 60 seconds.

### 6. update_learning_state/

**Route:** POST `/api/update-learning-state`

**Input body:**
```json
{
  "vendor_mappings": [
    { "dirty": "Caterpilalr Inc", "canonical": "Caterpillar Inc", "id": "SUP-00001" },
    { "dirty": "KOMATSU AUST P/L", "canonical": "Komatsu Australia Pty Ltd", "id": "SUP-00002" }
  ],
  "abbreviations": [
    { "abbrev": "HYDR", "expansion": "hydraulic" },
    { "abbrev": "RPLCMNT", "expansion": "replacement" }
  ],
  "classification_examples": [
    { "description": "Replacement hydraulic hose assembly Cat 789", "l1": "Equipment & Parts", "l2": "Mobile Equipment", "l3": "Haul truck parts", "verified": true }
  ],
  "instructions_append": "When vendor name contains 'WesTrac', always check if the equipment described is Caterpillar — WesTrac is the Cat dealer for WA."
}
```

**Logic:**
1. Read existing vendor_dictionary.json from SharePoint
2. Merge new vendor_mappings (deduplicate by dirty name)
3. Write updated vendor_dictionary.json back to SharePoint
4. Same for abbreviation_dictionary.json
5. Same for few_shot_examples.json
6. If instructions_append is provided, read agent_instructions.md from SharePoint, append the new instruction under a "## Learned Rules" section, write back
7. Return counts of new entries added

**Output body:**
```json
{
  "updated": true,
  "new_vendor_mappings": 2,
  "new_abbreviations": 2,
  "new_examples": 1,
  "instructions_updated": true,
  "total_vendor_dictionary_size": 347,
  "total_abbreviation_dictionary_size": 89,
  "total_examples_size": 156
}
```

**Timeout:** 30 seconds.

**This is the function that makes the agent self-improving.** Every run generates new vendor mappings, abbreviation expansions, and classification examples. The next run benefits from them.
